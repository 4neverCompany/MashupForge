# MMX OAuth 404 — Upstream Endpoint Down

**Date:** 2026-04-30
**Severity:** P1 — credential setup blocked for users without an existing API key
**Triage:** Developer
**Task id:** MMX-OAUTH-404-FIX

## Symptom

When a user runs `mmx auth login` (default `--method oauth`) — either
through their own terminal or via MashupForge's "or sign in via
terminal (OAuth)" link in Settings → AI Agent — the CLI prints a URL
to open in a browser:

```
https://platform.minimax.io/oauth/authorize?client_id=mmx-cli&...
```

That URL **returns HTTP 404** from MiniMax's server. The OAuth handshake
never completes, no token is written to `~/.mmx/config`, and the user
is stuck on the "Not Authenticated" state. From the user's perspective:
"OAuth 404 — credentials not persisting."

## Diagnosis

The 404 is upstream. We do not control `platform.minimax.io`. The
`mmx-cli` package shells out to that URL as part of its OAuth flow;
the URL is hard-coded in their CLI, the `client_id=mmx-cli` value is
their identifier, and the endpoint that's supposed to accept it on
their server is currently unreachable.

`mmx auth login` exposes two methods (per `mmx auth login --help`):

```
--method <method>   Auth method: oauth (default), api-key
--api-key <key>     API key to store
```

`--method api-key` writes the credential to local config without
hitting any OAuth endpoint. We already use that path in
`/api/mmx/setup` when the user pastes a key into the Settings input.
**That path works end-to-end and is unaffected by this incident.**

## What we changed (in our control)

`components/SettingsModal.tsx`:

- Removed the "or sign in via terminal (OAuth)" secondary link from
  the unauthenticated MMX setup block. The link called
  `handleMmxSetup()` which invoked `mmx auth login` server-side;
  every click since the upstream regression dead-ended at the 404.
- Replaced it with an external link to `https://platform.minimax.io/`
  so users who don't yet have an API key can sign up / locate their
  key and paste it into the existing Settings input.
- Pruned the now-unused `mmxOauthLabel` / `mmxNeedsInstall` constants.
- Updated the inline comment on `mmxSetupBlock` so the next person
  who reads the block isn't confused by the missing OAuth link.

The "Open MMX CLI to change provider/model" link in the
authenticated-state panel is unchanged — it drops the user into a
tmux session that runs `mmx auth status` (which passes for an
already-authenticated user) and then `exec bash -i`, never invoking
`mmx auth login`. So users who got in via API-key paste can still
reach the CLI for `mmx config set provider/model` configuration.

## What we did NOT change (out of our control)

- The OAuth endpoint at `platform.minimax.io` — upstream bug.
- The `mmx-cli` package — we don't fork it. If MiniMax fixes the
  OAuth endpoint, the existing tmux flow will start working again
  with no code change on our side; we'd just re-add a link if/when
  that happens.

## Recommended user-facing message (already deployed)

> **Don't have an API key? Get one at platform.minimax.io →**

Plus the existing instruction next to the password input:

> Stored in your local mmx config; never sent to MashupForge servers.

## Follow-ups to consider

- File an issue with MiniMax (`github.com/MiniMax-AI/cli` or their
  support channel) referencing the 404 with the specific `client_id`
  query string. Upstream fix unblocks the OAuth path globally.
- If MiniMax restores the endpoint, decide whether to re-add the
  OAuth link or keep the simplified API-key-only flow as the canonical
  path. Most users with an API key in hand still finish faster via
  paste; the OAuth flow is mainly useful for sign-up.
