## Commit 2 вАФ `d535ce3` fix(pipeline): quota-aware continue-gen + calendar-analysed schedule fallback

### вЬЕ Bug 1: quota gate on auto-start вАФ SOUND

- `quotaAlreadyMet` is a boolean IIFE derived from `daemon.weekFillStatus.scheduledTotal + pendingApprovalTotal >= targetTotal`. It is in the `useEffect` dependency array, so the effect re-evaluates when the computed boolean changes.
- `weekFillStatus` is always synchronously computed via `useMemo` in `usePipelineDaemon.ts:963-971` (never null in production). The null guard (`if (!fill) return false`) handles only the test stub case.
- The guard appears at both the outer check (before setTimeout) and inside the timer callback вАФ consistent with all other guards in the same effect.
- No race condition: `weekFillStatus` is computed from settings (always available), so there's no async hydration window where a full quota could be missed on mount.
- `autoStartFiredRef` one-shot guard still works correctly вАФ the quota gate only matters on the first mount when `autoStartFiredRef.current = false`.

### вЬЕ Bug 2: calendar-analysed retry in findBestSlot вАФ SOUND

- `constrained` flag correctly captures the three narrowing options that caused the pipeline path to exhaust the 7-day horizon: `postsPerDay != null`, `fillMode === 'depth'`, `horizonDays < 14`.
- Relaxed retry strips those three constraints but preserves `platforms` and `caps` вАФ appropriate, since caps are user limits and platforms are targeting data, not horizon constraints.
- The absolute fallback (`tomorrow @ 19:00`) is still reached when even the relaxed pick returns empty (e.g., zero engagement data).
- Manual Auto Schedule button path is unaffected вАФ it never sets the constrained options so `constrained = false` and the retry branch is never entered.
